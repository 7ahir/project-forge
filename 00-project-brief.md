# Project Brief

## Question This File Answers

What problem is this portfolio solving, and what PM skill spine is it built on?

## In One Line

Design and run the product strategy, adoption motion, and operating system for an AI coding assistant so it becomes a trusted developer tool and a scalable enterprise product.

## Scenario

The business has a technically strong AI coding assistant and early market pull.

The real problem sits at the seam between three product realities:

1. Individual developers will only keep using the assistant if it improves real coding workflows.
2. Teams will only expand usage if trust, standards, and control improve rather than degrade.
3. Enterprises will only pay at scale if rollout feels governable, measurable, and worth the change.

If those seams are unmanaged, the organization starts paying recurring tax in the form of:

- flashy demos that do not become habits
- product noise from loosely connected features
- enterprise requests that distort the roadmap
- weak quality signals and trust gaps
- leadership confidence built on the wrong metrics

## North Star

Make the assistant the most trusted way for a software team to draft, edit, review, and improve code inside real development workflows.

## Why This Matters

AI coding products rarely fail because the output looks unimpressive in a demo.

They fail because:

- developers do not trust them on real tasks
- teams cannot see how to roll them out safely
- buying teams cannot connect usage to operational value
- product teams confuse model progress with customer progress

When that happens, the wrong answer is more feature velocity. The right answer is a sharper product system.

## Working Assumptions

These assumptions are explicit so the reader can judge the logic instead of guessing what I presumed:

- The core model quality is already credible enough to create interest.
- The harder product problem is consistent usefulness inside real developer workflows.
- Team trust and governance are not edge concerns; they are part of mainstream adoption.
- The product must serve both bottom-up pull and top-down enterprise expansion.
- Any metrics in this portfolio are illustrative or simulated.

## Illustrative Outcome Targets

These are not claimed company metrics. They are example targets I would use to govern the work.

| Outcome | Illustrative baseline | Illustrative target |
|---|---|---|
| Weekly active developers who complete at least one high-value task with the assistant | 100 index | +35% |
| Edit acceptance rate on assisted changes | 42% | 60% |
| Sessions ending in full abandonment after low-trust output | 100 index | -25% |
| Team workspaces expanding from pilot to standard use | 20% | 45% |
| Enterprise pilots with a clear value story and rollout plan | 30% | 75% |
